# DevGuard — AI/ML Pipeline Context for Claude Code

## ML Service Architecture

A Python FastAPI sidecar running on port 8001, called by the Node.js backend via HTTP.
Falls back to rules-based scoring if the sidecar is unavailable.

## Three ML Features

### Feature 1: Blast Radius Classifier

Fine-tuned CodeBERT (microsoft/codebert-base) that predicts GREEN/YELLOW/RED from a code diff + file paths.
Replaces the rules-based scorer when model confidence > 0.8.

Input:

- diff: unified diff string
- files: array of modified file paths

Output:

- tier: GREEN | YELLOW | RED
- confidence: 0.0 to 1.0
- all_scores: probability for each tier

Training data: synthetically generated via GPT-4o (50 batches x 10 samples = ~500 labeled diffs).
Labels: GREEN = safe changes, YELLOW = logic/dependency changes, RED = auth/payments/migrations/infra.

### Feature 2: Anomaly Detector

Autoencoder trained on embeddings of normal proposals (text-embedding-3-small, 1536-dim).
Flags proposals with high reconstruction error as anomalies — catches Codex going out of scope before humans see it.
Threshold = 95th percentile of training reconstruction errors.

Input:

- proposal: { summary, diff, files_to_modify }

Output:

- is_anomaly: boolean
- reconstruction_error: float
- anomaly_score: normalized float (0–3+, above 1.0 = anomaly)
- threshold: float

Effect on pipeline: if is_anomaly is true, force tier to RED and add warning to approval card.

### Feature 3: Semantic Similarity (Duplicate Detection)

Cosine similarity between new proposal embedding and stored historical proposal embeddings.
Prevents Codex from regenerating near-identical proposals for similar tickets.
Threshold: 0.92 similarity score.

Input:

- proposal: { summary, diff, files_to_modify }

Output:

- is_duplicate: boolean
- similarity_score: float
- matched_proposal_id: string | null

Effect on pipeline: if is_duplicate is true, skip generation and return the existing proposal with a warning.

## ML Sidecar Endpoints

```
POST /classify      # Feature 1 — CodeBERT classifier
POST /anomaly       # Feature 2 — Autoencoder anomaly detection
POST /similarity    # Feature 3 — Duplicate detection
GET  /health        # Model load status
```

## Where Each Feature Plugs Into the Main Pipeline

```
Ticket Input
    ↓
[POST /similarity] ← check for duplicates first, skip if match found
    ↓
RAG context retrieval
    ↓
Codex proposal generation
    ↓
[POST /anomaly] ← flag anomalous proposals, force RED if triggered
    ↓
[POST /classify] ← ML tier prediction, fallback to rules if confidence < 0.8
    ↓
Reviewer agent
    ↓
Merge risk scores → final tier
    ↓
Route: GREEN auto-execute | YELLOW/RED send approval card
```

## Fallback Behavior

- /classify unavailable or confidence < 0.8 → use rules-based scorer
- /anomaly unavailable → skip anomaly check, log warning
- /similarity unavailable → skip duplicate check, proceed normally
- Any ML error → fail safe to RED, never block the pipeline

## Models

- Blast radius classifier: microsoft/codebert-base fine-tuned, saved to models/blast_radius_classifier/
- Anomaly detector: custom PyTorch autoencoder, saved to models/anomaly_detector.pt
- Similarity: no saved model, computed at runtime using OpenAI embeddings + cosine similarity

## Training Data

- Classifier: ~500 synthetic labeled diffs generated by GPT-4o, split 80/10/10 train/val/test
- Autoencoder: embeddings of all GREEN-tier historical proposals (normal baseline)
- Similarity: embeddings of all stored proposals in the audit log (grows over time)

## ML Sidecar Tech Stack

- Python 3.11, FastAPI
- PyTorch, HuggingFace transformers
- OpenAI SDK (embeddings)
- scikit-learn (cosine similarity)
- Runs on port 8001
- Separate requirements.txt and Dockerfile from main Node.js backend
