# NexGuard — Project Context for Claude Code

## What We're Building

NexGuard is a human-governed AI coding agent orchestration platform built on top of OpenAI Codex. It intercepts every AI-generated code change, scores its blast radius, and routes it through a risk-tiered approval system before anything touches the codebase. It covers the full SDLC: development, testing, deployment, incident response, communication, and governance.

## Core Concept

Every Codex action is a _proposal_, not an execution. A risk engine scores each proposal and assigns a tier:

- GREEN: auto-execute, log only
- YELLOW: execute + notify with 1-hour veto window
- RED: hard block until explicit human approval

## Tech Stack

**Node.js Backend (port 3000)**
- Runtime: Node.js + TypeScript, Express
- ORM: Prisma with PostgreSQL
- Queue: BullMQ + Redis
- Notifications: Slack Webhooks + Slack Interactive Components
- GitHub Integration: Octokit
- Vector Store: ChromaDB (shared with ML sidecar)
- AI: OpenAI API (gpt-4o for proposals, text-embedding-3-small for RAG)

**Python ML Sidecar (port 8001)**
- Runtime: Python 3.11, FastAPI
- Models: PyTorch, HuggingFace transformers (CodeBERT)
- Similarity: scikit-learn cosine, OpenAI embeddings
- Called by Node.js backend via HTTP; falls back to rules-based scoring if unavailable

**Frontend Dashboard:** Next.js
**Infra:** Docker Compose

## ML Pipeline (see context/CODE_AGENT.md for full details)

```
Ticket Input
    ↓
[/similarity] ← Feature 3: duplicate check, skip if match found (threshold 0.92)
    ↓
RAG context retrieval
    ↓
Codex proposal generation (gpt-4o)
    ↓
[/anomaly]   ← Feature 2: autoencoder anomaly check, force RED if triggered
    ↓
[/classify]  ← Feature 1: CodeBERT tier prediction, fallback to rules if confidence < 0.8
    ↓
Reviewer agent (gpt-4o second pass)
    ↓
Merge scores → final tier
    ↓
GREEN auto-execute | YELLOW/RED → Slack approval card
```

- `/classify` — Fine-tuned CodeBERT (microsoft/codebert-base), saved to `ml/models/blast_radius_classifier/`
- `/anomaly`  — PyTorch autoencoder on GREEN-proposal embeddings, saved to `ml/models/anomaly_detector.pt`
- `/similarity` — Runtime cosine similarity via ChromaDB + text-embedding-3-small (no saved model)
- All ML errors fail safe to RED and never block the Node.js pipeline

## Prisma Schema (prisma/schema.prisma)

```prisma
generator client {
  provider = "prisma-client-js"
}

datasource db {
  provider = "postgresql"
  url      = env("DATABASE_URL")
}

model Proposal {
  id                    String    @id @default(uuid())
  ticketTitle           String
  ticketDescription     String
  repo                  String
  reporter              String
  summary               String
  diff                  String    @db.Text
  filesToModify         String[]
  risks                 String[]
  confidence            Float
  whatIDidntDo          String
  testCoverageAffected  Boolean
  tier                  Tier
  riskScore             Int
  riskReasons           String[]
  failSafeTriggered     Boolean   @default(false)
  status                ProposalStatus @default(PENDING)
  createdAt             DateTime  @default(now())
  updatedAt             DateTime  @updatedAt

  approval              Approval?
  auditLogs             AuditLog[]
}

model Approval {
  id          String    @id @default(uuid())
  proposalId  String    @unique
  proposal    Proposal  @relation(fields: [proposalId], references: [id])
  actor       String
  action      ApprovalAction
  reason      String?
  createdAt   DateTime  @default(now())
}

model AuditLog {
  id          String      @id @default(uuid())
  proposalId  String?
  proposal    Proposal?   @relation(fields: [proposalId], references: [id])
  event       AuditEvent
  actor       String      # "codex" or human username
  metadata    Json?
  createdAt   DateTime    @default(now())
}

model Incident {
  id          String    @id @default(uuid())
  description String
  logs        String    @db.Text
  repo        String
  reporter    String
  status      IncidentStatus @default(ACTIVE)
  createdAt   DateTime  @default(now())
  resolvedAt  DateTime?

  hotfixes    Hotfix[]
}

model Hotfix {
  id          String    @id @default(uuid())
  incidentId  String
  incident    Incident  @relation(fields: [incidentId], references: [id])
  summary     String
  diff        String    @db.Text
  confidence  Float
  blastRadius String    # LOW, MEDIUM, HIGH
  status      HotfixStatus @default(PENDING)
  appliedBy   String?
  appliedAt   DateTime?
  createdAt   DateTime  @default(now())
}

enum Tier {
  GREEN
  YELLOW
  RED
}

enum ProposalStatus {
  PENDING
  AUTO_EXECUTED
  AWAITING_APPROVAL
  APPROVED
  REJECTED
  VETOED
  EXECUTED
  FAILED
}

enum ApprovalAction {
  APPROVED
  REJECTED
  VETOED
}

enum AuditEvent {
  PROPOSED
  AUTO_EXECUTED
  APPROVAL_SENT
  APPROVED
  REJECTED
  VETOED
  EXECUTED
  FAILED
  INCIDENT_DECLARED
  HOTFIX_APPLIED
  ARTIFACT_GENERATED
}

enum IncidentStatus {
  ACTIVE
  RESOLVED
}

enum HotfixStatus {
  PENDING
  APPLIED
  REJECTED
}
```

## Example Prisma Usage Patterns

### Writing a proposal

```typescript
const proposal = await prisma.proposal.create({
  data: {
    ticketTitle: ticket.title,
    ticketDescription: ticket.description,
    repo: ticket.repo,
    reporter: ticket.reporter,
    summary: codexOutput.summary,
    diff: codexOutput.diff,
    filesToModify: codexOutput.files_to_modify,
    risks: codexOutput.risks,
    confidence: codexOutput.confidence,
    whatIDidntDo: codexOutput.what_i_didnt_do,
    testCoverageAffected: codexOutput.test_coverage_affected,
    tier: riskScore.tier,
    riskScore: riskScore.score,
    riskReasons: riskScore.reasons,
    failSafeTriggered: riskScore.failSafeTriggered,
    status: "PENDING",
  },
});
```

### Writing an audit log entry

```typescript
await prisma.auditLog.create({
  data: {
    proposalId: proposal.id,
    event: "PROPOSED",
    actor: "codex",
    metadata: {
      tier: riskScore.tier,
      score: riskScore.score,
    },
  },
});
```

### Approving a proposal

```typescript
await prisma.$transaction([
  prisma.approval.create({
    data: {
      proposalId: id,
      actor: approver,
      action: "APPROVED",
    },
  }),
  prisma.proposal.update({
    where: { id },
    data: { status: "APPROVED" },
  }),
  prisma.auditLog.create({
    data: {
      proposalId: id,
      event: "APPROVED",
      actor: approver,
    },
  }),
]);
```

### Fetching audit feed for dashboard

```typescript
const feed = await prisma.auditLog.findMany({
  orderBy: { createdAt: "desc" },
  take: 50,
  include: {
    proposal: {
      select: {
        ticketTitle: true,
        tier: true,
        repo: true,
      },
    },
  },
});
```

## Key Implementation Rules

1. Always use prisma.$transaction() when writing to multiple tables atomically
2. Audit log is append-only — never update or delete AuditLog records
3. Blast radius scorer always defaults to RED if it throws any exception
4. All external API calls (OpenAI, GitHub, Slack) must have a mock fallback
5. Incident mode countdown timer is 45 seconds
6. Deployment approval cards require checkbox acknowledgment before Approve is enabled
7. Communication agent always reads from AuditLog — never from live state
8. Use Prisma Studio during demo to show live DB state to judges

## What Is Out of Scope

- Real Jira integration (mock /intake endpoint)
- ML-trained blast radius classifier (rules-based only)
- Approval pattern learning
- Multi-repo support
- Dashboard authentication
